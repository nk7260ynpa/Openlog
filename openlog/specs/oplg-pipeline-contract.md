# Spec: `/oplg:*` three-stage pipeline contract

- **Status:** Active
- **Owner:** Openlog maintainers
- **Last updated:** 2026-05-03
- **Applies to:** `/oplg:explore`, `/oplg:apply`, `/oplg:record` (skills and
  slash commands generated by `openlog init`).

## Purpose

This spec is the single source of truth for **what each `/oplg:*` workflow
is allowed and required to do**, and for **how they hand off to each
other**. The individual workflow templates under
`src/core/templates/workflows/*.ts` implement this contract; this document
defines the contract they implement, so that future edits to those
templates can be checked for divergence.

The pipeline is three sequential stages, each with a strict role:

```
┌──────────────────┐   handoff block    ┌─────────────────┐   diff/commit    ┌──────────────────┐
│  /oplg:explore   │ ─────────────────▶ │  /oplg:apply    │ ───────────────▶ │  /oplg:record    │
│  (read-only)     │   (paste-ready)    │  (edit + ship)  │   (no README)    │  (log changes)   │
└──────────────────┘                    └─────────────────┘                  └──────────────────┘
```

Each arrow is a **handoff**, not a transition the workflow performs by
itself. The user (or an automation) chooses when to advance to the next
stage. Workflows must not chain into one another implicitly.

## Stage 1 — `/oplg:explore <topic>` (research)

### Role
Map out the relevant code/docs for a question or proposed change. Produce a
structured, paste-ready report. **Make no changes.**

### MUST
- Restate the topic in one sentence so the user can confirm intent before
  evidence-gathering begins.
- Cite every concrete claim about the codebase with `file_path:line_number`.
- Output the fixed sections, in this order:
  1. `Summary` — 2–4 sentences, lead with the conclusion.
  2. `Key files` — bullet list of `file:line` with one-line role.
  3. `How it works` — short narrative walking through the current state.
  4. `Gaps, risks, or open questions`.
  5. `Suggested next actions for /oplg:apply`.
  6. `Handoff block` — one or more paste-ready lines beginning with
     `/oplg:apply`, each describing a self-contained edit.
- If the topic is empty or extremely vague, call `AskUserQuestion` to
  clarify before doing any work.

### MUST NOT
- Call `Edit`, `Write`, `NotebookEdit`, or any tool that mutates files.
- Run mutating shell commands: `git add/commit/push/checkout/reset/stash/
  mv/rm/tag/merge/rebase/pull`, `git fetch --prune`, package installs
  (`npm i`, `pnpm i`, `pnpm run build`, `pnpm run dev*`), database
  migrations, container restarts, or anything that touches `dist/`,
  `node_modules/`, the database, or external services.
- Invent file paths, function names, or commit hashes. If something is
  unverified, label it as such and explain how the user could verify.
- Begin editing if the user says "just go ahead" — instead, stop and tell
  them to re-run as `/oplg:apply <action>` using the handoff block.

### Read-only commands that ARE allowed
`git log`, `git status`, `git diff`, `git show`, `git blame`, `ls`, `find`,
`tree`, `grep`/`rg`, `node --version`, plus `Read`, `Glob`, `Grep`,
`WebFetch`, `WebSearch`.

### Output handoff to Stage 2
The `Handoff block` is the contract. Each line is a self-contained
`/oplg:apply` invocation, and each line should map to **exactly one
commit** when Stage 2 runs.

## Stage 2 — `/oplg:apply <action>` (edit and ship)

### Role
Translate an action description (typically pasted from Stage 1's handoff
block) into concrete edits, locally verify, and ship one commit per entry.

### MUST
- Treat the action description as authoritative. If empty or extremely
  vague, call `AskUserQuestion` to clarify before editing.
- Run `git status --short` and `git diff --stat` first to avoid clobbering
  uncommitted work.
- For each entry (sub-task) in the action, loop through:
  1. Edit the relevant files (minimal scope; no incidental refactors).
  2. Update `README.md` whenever the entry changes user-facing CLI flags,
     install steps, public APIs, configuration, or dependencies. Skip
     explicitly with a "no README change needed" note for internal-only
     entries.
  3. Run the project's fastest local check (this repo: `node build.js` or
     `pnpm run build` — there is no test runner).
  4. Stage source + README + rebuilt artifacts (e.g. `dist/`), commit
     using Conventional Commits, and `git push` to the existing upstream.
- One entry → one commit → one push. Never batch unrelated entries.

### MUST NOT
- Use destructive git operations without explicit user instruction:
  `reset --hard`, `push --force`, `push --force-with-lease`, branch
  deletion, history rewrites.
- Skip `README.md` for user-facing changes. README is the public contract
  for this repo.
- Leave `TODO` / `FIXME` / `HACK` markers in the final commit, unless the
  entry itself is to add such a marker.
- Modify `CLAUDE.md`, `settings.json`, or other docs unless the entry
  explicitly targets them.
- Modify files under `openlog/changes/` — that surface is owned by
  Stage 3.

### Handoff to Stage 3
After commits land, Stage 3 (`/oplg:record`) inspects `git log` plus the
working tree to write the corresponding records. Stage 2 does not write
anything under `openlog/changes/` itself.

## Stage 3 — `/oplg:record` (log changes)

### Role
Turn the current session's commits and uncommitted edits into one record
file per logically distinct entry under `openlog/changes/`, and sync
internal docs that have drifted.

### MUST
- Auto-derive title and slug from the diff itself; ignore any text the user
  types after the command (treat it only as supplementary context).
- One record per logically distinct entry; never merge unrelated entries.
- Filename pattern: `openlog/changes/<YYYY-MM-DD>_<NN>-<slug>.md`, where
  `<NN>` is a per-day completion-order counter starting at `_01`. If a
  same-named file already exists, pick the next free `<NN>`; never
  overwrite.
- Use the standard skeleton (Summary, Motivation / context, Key changes,
  Impact, Verification, Follow-ups). Content must come from real diffs,
  commits, or conversation facts — never fabricated.
- Sync internal docs when applicable: `openlog/project.md`,
  `openlog/specs/`, `CHANGELOG.md` (if present), team-shared `CLAUDE.md`
  (only when the change affects AI collaboration workflows). Use minimal
  diffs.
- Commit (typically `docs(openlog): ...`) and push.

### MUST NOT
- Modify `README.md` — that surface is owned by Stage 2.
- Modify the user's global `~/.claude/CLAUDE.md`.
- Fabricate code changes or test results to fill the skeleton.
- Use destructive git operations without explicit user instruction.

### Handoff back to the user
The summary report at the end of Stage 3 is the closing artifact of the
pipeline cycle. The user can then run another `/oplg:explore` for the next
topic.

## Cross-stage invariants

These properties hold across the pipeline; any workflow change that breaks
one of them violates this spec.

1. **Surface ownership is exclusive.**
   - `README.md` → owned by Stage 2 only.
   - `openlog/changes/` → owned by Stage 3 only.
   - `dist/` → produced by Stage 2's local verification step (this repo
     commits `dist/` since v0.3.5, per `CLAUDE.md`).
2. **Mutating tools are gated by stage.** Stage 1 is read-only. Stages 2
   and 3 may use `Edit`, `Write`, `git add/commit/push`, but never
   destructive git verbs without explicit user instruction.
3. **Each entry maps to one commit in Stage 2 and one record in Stage 3.**
   If Stage 2 produced N commits for N entries, Stage 3 produces N record
   files (modulo entries that were already recorded earlier).
4. **Handoff is data, not control flow.** Each stage emits a structured
   artifact (handoff block, commits, record files) and stops. No stage
   automatically invokes the next.
